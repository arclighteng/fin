4) Habitual spend is treated as “recurring” (inflates baseline, misleads fixed-cost picture)

Severity: High (truthfulness + budgeting decisions)
Where:

Habitual flag creation: src/fin/classify.py:447 sets cadence_label="habitual" when len(items) >= 6

Bucket decision: src/fin/classify.py:676 treats pattern.is_recurring as recurring (and is_recurring = subscription OR habitual)

Impact / behavior

Groceries/gas/Amazon-type “habitual” spending gets rolled into recurring_cents, which drives:

baseline_cents = income - recurring

net_cents = income - recurring - one_off

Users read “recurring” as fixed obligations, but your math includes variable life spend.

Repro

Ensure a merchant appears 6+ times in lookback (default 400 days) with negative amounts.

Run month summary / anything using summarize_month().

That merchant’s spend shows up as recurring (even if irregular).

Fix

Split buckets: subscription_recurring vs habitual_variable, and compute baseline using only subscriptions/utilities if that’s what you want baseline to mean.

6) Filtered views still use patterns learned from all accounts (recurring/discretionary can be wrong)

Severity: Medium (misclassification in filtered dashboards)
Where: src/fin/analysis.py

analyze_periods: patterns = _detect_patterns(conn, lookback_days=800) at 498

analyze_custom_range: patterns = _detect_patterns(conn, lookback_days=800) at 655
Neither passes account_filter.

Impact / behavior

If you filter the dashboard to a subset of accounts, the recurring classification still reflects merchants’ history across every account.

Example: A merchant is recurring on Account A (credit card) but you filter to Account B (checking) where it only occurred once—your filtered view can still classify it as recurring.

Repro

Same merchant appears regularly on one account, rarely on another.

Filter dashboard to the “rare” account.

Recurring totals can still reflect the “regular” account’s history.

Fix

When account_filter is provided, pass it to _detect_patterns(..., account_filter=account_filter).

7) Pending transactions are included in totals and alerts (numbers can swing / be “wrong” vs posted reality)

Severity: Medium (truthfulness + user trust)
Where: all major analysis queries omit pending filtering:

_analyze_single_period query: src/fin/analysis.py:353 (WHERE posted_at >= ? AND posted_at < ?)

analyze_custom_range query: src/fin/analysis.py:672 (WHERE posted_at >= ? AND posted_at <= ?)

classify_month query: src/fin/classify.py in classify_month() (no pending=0 clause)

Impact / behavior

A pending charge that later drops or changes amount can appear in “spend” today and disappear later, creating “moving target” totals.

Make it explicit and consistent:

default exclude pending=1 in financial totals (and optionally show a “pending” section), OR

clearly label that totals include pending and show a pending subtotal.

8) Transfer drill-down is incomplete: outgoing transfers don’t show up in transfer_items

9) Keyword-based transfer detection is over-broad → real expenses can be incorrectly excluded

Severity: Medium (can materially distort spend)
Where:

src/fin/classify.py:319+ _is_transfer() includes broad keywords like "autopay", "bill pay", "credit card" etc.

src/fin/analysis.py:621+ _is_cc_payment_expense() includes "amex" which can match non-payment strings.

Impact / behavior

Merchants/descriptions containing these substrings can be classified as transfers and removed from expense totals.

Example risk strings:

"... AUTOPAY ..." for utilities/rent

"... BILL PAY ..." used by banks for a wide variety of payees

"amex" appearing in contexts that aren’t actually a payment record

Fix

Tighten patterns to more payment-specific phrases (e.g., "payment to", "card payment", "statement payment", "autopay payment"), and/or require corroborating signals (account type + known institution names + amount ranges).
